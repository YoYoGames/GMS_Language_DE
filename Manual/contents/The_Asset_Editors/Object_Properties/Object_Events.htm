Objekt-Ereignisse

Was sind also Objekt-Events? Im Grunde sind dies diskrete Momente in der Spielschleife, in denen Dinge auf der Grundlage dessen, was Sie für sie programmiert haben, geschehen sollen. GameMaker Studio 2 arbeitet mit Zyklen dieser Ereignisse - von dem Moment, in dem ein Von dem Moment an, in dem ein Raum gestartet wird, bis zu dem Moment, in dem er beendet wird, läuft eine Spielschleife, in der bei jedem Schritt eine Reihe von Ereignissen ausgeführt oder überprüft wird, und Sie können wählen, ob Sie Code oder DnD™-Aktionenin Ihren Objekten platzieren, die auf diese Ereignisse reagieren (ein Schritt ist ein Moment in der Spielzeit, der durch die Einstellung der Raumgeschwindigkeit bestimmt wird und auch als Frame bezeichnet werden kann) .

Diese Optionen sind:

  • Ereignis hinzufügen - Fügt ein neues Ereignis aus der Ereignisliste hinzu.
  • Ausschneiden / Kopieren / Einfügen von Events - Ausschneiden, Kopieren oder Einfügen des ausgewählten Events. Sie können auch die Standard-Tastaturkürzel verwenden: / + X, / + C und / + V.

  • In Drag and Drop / Code konvertieren - Objekt-Events werden mit der Methode erstellt, die Sie beim Erstellen des Projekts gewählt haben, entweder mit Code oder mit Drag and Drop™-Aktionen. Mit dieser Option können Sie von der einen zur anderen Methode konvertieren, unabhängig des Projekttyps.

  • die, die jeden einzelnen Spielschritt ausführen
  • solche, die durch ein Spielereignis "ausgelöst" werden, wie z. B. das Erreichen der Raumkante durch die Instanz oder einen Tastatur- oder Mausdruck.

Zerstören

Dieses Ereignis ist das Ereignis, das ausgeführt wird, wenn eine Instanz zerstört wird. Es wird beim Hinzufügen von Verhaltensweisen zu Objekten oft übersehen, kann aber sehr nützlich sein, z. B. durch das Erzeugen von Explosions- oder Partikeleffekten, wenn ein Feind getötet wird, oder für das Re-Spawnen eine neue Instanz des Objekts in einem anderen Teil des Raums oder sogar zum Hinzufügen von Punkten zu einem Spielstand.

Aufräumen

Dieses Ereignis wird nach jedem Ereignis aufgerufen, das eine Instanz des Objekts aus dem Raum entfernt. Es wird also ausgelöst, wenn:

  • die Instanz wird zerstört
  • der Raum endet
  • das Spiel endet

Es ist für Sie gedacht, um alle dynamischen Ressourcen zu "bereinigen", die Sie möglicherweise in Ihrem Spiel haben (wie Oberflächen, Datenstrukturen, etc...) oder um jede Aufgabe auszuführen, die Sie einmalig benötigen, wenn die Instanz in irgendeiner Weise aus dem Spiel entfernt wird.

Beachten Sie, dass dieses Ereignis sofort nach dem Ereignis, das es ausgelöst hat, aufgerufen wird, die Instanz aber erst nach dem Ende des aktuellen Ereignisses tatsächlich aus dem Spiel entfernt wird. Wenn Sie z. B. instance_destroy() im Step-Event aufrufen, wird das Destroy-Event aufgerufen, dann das Clean Up-Event, und dann wird der Rest des Step-Events beendet. Das bedeutet, dass jeglicher Code, den Sie nach dem Aufruf von instance_destroy() haben, noch ausgeführt wird und eine potenzielle Fehlerursache darstellt, wenn Sie eine Datenstruktur oder eine andere Ressource, die der Code benötigt, aufgeräumt haben, daher ist bei der Verwendung dieses Ereignisses Vorsicht geboten.

Alarm

Die Alarmkategorie ist in 12 Ereignisse unterteilt, eines für jeden der möglichen Alarme, die in einer Instanz eingestellt werden können. Wenn Sie also auf die Kategorie " Alarm hinzufügen" klicken, wird dieses Fenster angezeigt:

Hier wählen Sie den Alarm aus, den Sie erstellen möchten, und sobald dies geschehen ist, sehen Sie, dass er dem Ereignisfenster hinzugefügt wurde. Fenster hinzugefügt wurde, so dass Sie ihm wie gewohnt Code hinzufügen können. Aber was ist ein Alarm? Nun, es ist ein spezielles Ereignis, das nichts tut, es sei denn, der Alarm wurde vorher gesetzt, und dann wird gewartet, bis der Alarm auf 0 heruntergezählt ist, bevor die Aktionen oder der Code ausgeführt werden, die Sie hinzugefügt haben oder Code ausführt, den Sie ihm hinzugefügt haben.

Wenn der Alarm 0 erreicht hat und der Code ausgeführt wird, zählt er dann auf -1 zurück, wo er bleibt, bis er wieder gesetzt wird (das heißt, Sie können den Wert eines Alarms überprüfen, um zu sehen, ob er größer als -1 ist, was Ihnen sagt, ob er läuft oder nicht). Also, sagen wir Sie setzen alarm[0] im Ereignis "create" des Objekts auf 30, bedeutet dies, dass GameMaker Studio 2 30 Spielschritte herunterzählt, bevor es die Aktionen oder den Code ausführt, die im Ereignis alarm[0] platziert sind. Beachten Sie, dass das Setzen eines Alarms auf 0 nicht dazu führt, dass der Alarmcode ausgeführt wird, da das Ereignis zwar ausgelöst wird, aber der Alarm sofort auf -1 gesetzt wird, so dass der Code übersprungen wird. Wenn Sie einen Alarm benötigen, um gleich den nächsten Schritt auszuführen, dann sollten Sie ihn auf 1 setzen.

Dies kann sehr nützlich sein, da es Ihnen erlaubt, Dinge zu bestimmten Zeitpunkten in Bewegung zu setzen, und Sie können sie sogar wiederholen lassen, da nichts dagegen spricht, einen Alarm in seinem eigenen Ereignis zu setzen. Stellen Sie sich vor, Sie haben ein Monster und wollen, dass es sich alle drei Sekunden nach rechts dreht. Sie würden einen Alarm in seinem Erstellungsereignis auf die Raumgeschwindigkeit * 3 setzen (wenn die Raumgeschwindigkeit 30 ist, sind das 30 Schritte pro Sekunde, also multiplizieren Sie das mit 3 und Sie erhalten 3 Sekunden!) und dann würden Sie im Alarmereignis den Code oder die Aktion haben, um seine Richtung einzustellen, sowie Aktion, um die Richtung einzustellen, sowie die Aktion (oder den Code), um den Alarm wieder auf Raumgeschwindigkeit * 3 zu setzen. Auf diese Weise können Sie einfache Spielschleifen einrichten, in denen Dinge nur in bestimmten Intervallen passieren.

Es ist erwähnenswert, dass ein Alarm, der keine Aktionen oder Code enthält, nicht herunterzählt. Aber auch mit nur einem Kommentar und ohne Code oder Aktionen, dann wird der Alarm weiter herunterzählen und kann wie gewohnt eingestellt und überprüft werden.

Schritt

GameMaker Studio 2 unterteilt die Zeit in Schritte , wobei die Raumgeschwindigkeit definiert, wie viele dieser Schritte es pro Sekunde geben soll (ein Schritt kann auch als Frame bezeichnet werden). Ein einzelner Schritt, ist im Grunde die Schleife, die ständig mit allen Ereignissen, die geprüft und bei Bedarf ausgelöst werden, während das Spiel läuft. Wie Sie sich also vorstellen können, ist das Step-Ereignis ein Ereignis, das bei jedem einzelnen Schritt des Spiels geprüft wird, solange die Instanz existiert.

Das Step-Ereignis besteht eigentlich aus drei Unterereignissen, die im Folgenden beschrieben werden:

Für die meisten Dinge ist das Standard-Schritt-Ereignis gut geeignet, aber manchmal möchten Sie ein wenig mehr Kontrolle darüber haben, welcher Code zu welchem Zeitpunkt ausgeführt wird. Zu diesem Zweck gibt es die Step-Ereignisse Begin und End . Alle drei werden bei jedem Schritt überprüft, aber ihre Reihenfolge wird sich nie ändern, selbst wenn zukünftige Updates der GameMaker Studio 2-Engine andere Ereignisse ändern, was bedeutet, dass dies die einzige zuverlässige Methode ist, um sicherzustellen, dass etwas immer vor etwas anderem passiert.

Wofür kann das Schritt-Ereignis verwendet werden? Nun, es kann für Aktionen oder Code verwendet werden, der kontinuierlich ausgeführt werden muss. Wenn zum Beispiel ein Objekt einem anderen folgen soll, können Sie hier die Bewegungsrichtung in Richtung des Objekts, dem wir folgen, anpassen anpassen, damit es sich gleichmäßig hinterher bewegt. Seien Sie jedoch vorsichtig mit diesem Ereignis und packen Sie nicht viele komplizierte Aktionen in das step-Ereignis von Objekten, vor allem, wenn Sie planen, viele Instanzen des Objekts in Ihrem Spielraum zu haben, denn das könnte das Spiel verlangsamen. Viele Dinge können in Alarmen untergebracht werden oder so eingestellt werden, dass sie mit einigen der anderen Ereignisse ausgelöst werden, anstatt ständig zu passieren.

Kollision

Beim Erstellen eines Spiels ist es natürlich sehr wichtig, dass Sie wissen, wann zwei (oder mehr) Instanzen eines Objekts kollidiert sind, und dafür haben wir das Kollisionsereignis. Dies ist ein Ereignis, das Sie in einem Objekt platzieren und dann angeben, gegen welches andere Objekt auf Kollisionen überprüft werden soll.

Wenn Sie die Physik nicht eingeschaltet haben, werden diese Kollisionen auf der Grundlage der Maske der beiden Objekte berechnet (die Maske wird in den Sprite-Eigenschaften definiert oder kann unabhängig in den Objekteigenschaften zugewiesen werden) und ob sie sich überschneiden oder nicht. Beachten Sie, dass, wenn eine der beiden Instanzen in der Kollision keine Maske zugewiesen hat (oder die Sprite-Maske auf nichts eingestellt ist), selbst wenn sie etwas zeichnet, keine Kollisionen erkannt werden.

Wenn Sie Physik eingeschaltet haben, basiert die Kollision auf dem Typ der Kollisionsform(Fixture), den Sie für das Objekt in seinen Physikeigenschaften definiert haben, ebenso wie seine Reaktion auf die Kollision. Diese bedeutet, dass Sie möglicherweise keinen Code benötigen, um die Kollision zu behandeln, aber dieses Ereignis muss trotzdem mindestens einen Kommentar enthalten , damit die Kollisionen erkannt werden.

Schließlich ist zu beachten, dass alle Kollisionen einmal pro Spielschritt berechnet werden, bevor das Kollisionsereignis ausgelöst wird, so dass beim Ausführen des Kollisionsereignisses alle Kollisionen bereits berechnet und vorbelegt sind. Das bedeutet, dass wenn Sie eine Instanz in diesem Ereignis erstellen und dann versuchen, eine Kollision damit zu überprüfen, wird die Kollision erst bei der nächsten Iteration der Spielschleife erkannt oder aufgelöst.

Tastatur, Tastatur drücken, Tastaturfreigabe

Es ist sehr wichtig, dem Spieler die Möglichkeit zu geben, die verschiedenen Aspekte Ihres Spiels zu steuern, und zu diesem Zweck bietet Ihnen GameMaker Studio 2 eine sehr umfassende Liste von Tastaturereignissen, die in jeder der drei Hauptkategorien der Tastatur verwendet werden können. Für die allgemeine Tastaturkategorie wird es kontinuierlich bei jedem Schritt ausgelöst, solange die ausgewählte Taste gedrückt ist, während die Ereignisse der Kategorie "Drücken und Loslassen" nur einmal ausgelöst werden, wenn die Taste anfangs gedrückt oder losgelassen wird.

Es ist zu beachten, dass Tastaturereignisse tatsächlich in allen aktiven Instanzen in einem Raum ausgelöst werden, wenn eine Taste betätigt wird, aber nur die Instanzen reagieren, für die ein Ereignis für diese bestimmte Taste definiert ist, und Sie können mehrere Tastaturereignisse in Sie können mehrere Tastaturereignisse in einem beliebigen Objekt erstellen, und die Instanzen dieses Objekts reagieren auf alle, während das Spiel läuft.

Maus

Die Kategorie "Maus" ist in eine Reihe von Ereignissen unterteilt, die ausgewählt werden können, um Ihnen eine genauere Kontrolle darüber zu geben, was in Ihrem Spiel passiert. Hier können Sie genau sehen, was diese Ereignisse sind:

Die Tastenereignisse links , rechts und mitte (egal ob normal, gedrückt oder losgelassen) wirken alle wirken auf die Maske der Instanz, die das Ereignis hat. Das bedeutet, dass GameMaker Studio 2 die Position der Maus im Raum, wenn diese Tasten betätigt werden, mit den Kollisionsmasken der Instanzen vergleicht, die ein Mausereignis haben. Wenn Wenn es eine "Kollision" mit dem Begrenzungsrahmen der Instanz gibt , wird das Ereignis ausgelöst. Stellen Sie also sicher, dass jede Instanz mit diesen Ereignissen ein Sprite mit einer gültigen Kollisionsmaske hat Maske hat oder dass das Objekt in den Objekteigenschaften ein Maskensprite ausgewählt hat. Wie ihre Namen andeuten, werden diese Ereignisse entweder einmalig ausgelöst, wenn die gewählte Maustaste zum ersten Mal gedrückt oder losgelassen wird, oder kontinuierlich bei jedem Schritt, während die Taste beibehalten wird.

Die Mauseintritts- und -austrittsereignisse ähneln den Tastenereignissen insofern, als dass sie ebenfalls von der Maske der Instanz abhängen, aber diesmal werden sie ausgelöst, wenn die Maus die Instanz zum ersten Mal "betritt" (berührt) oder wenn die Maus sie "verlässt" (aufhört, die Instanz zu berühren). Diese Ereignisse sind jedoch nicht kontinuierlich, sondern werden nur einmal ausgelöst, wenn die Maus das Objekt betritt oder verlässt - sie sind also eine ideale Methode, um z. B. Schaltflächen zu erstellen, die sich ändern müssen, wenn die Maus Sie sind daher ideal für Schaltflächen, die sich ändern müssen, wenn die Maus über sie fährt, und die sich wieder normalisieren, wenn die Maus entfernt wird.

Schließlich haben wir noch einen weiteren Bereich zu den Mausereignissen, der sich Global Mouse nennt. In diesem Untermenü finden Sie eine Auswahl von Ereignissen, die zur Aufzeichnung von Mausereignissen in Instanzen dienen, auch wenn sich die Maus nicht über ihnen oder sogar in ihrer Nähe ist. Dies sind Ereignisse, die für alle Instanzen generiert werden, und wenn für das angegebene Ereignis Aktionen oder Code definiert sind, werden diese ausgeführt, unabhängig von der Position der Maus innerhalb des Spielraums.

Bitte beachten Sie, dass bei mobilen Geräten oder Geräten mit Touchscreen die linke Maustaste auch zur Prüfung auf einen Fingertab auf einem Touchscreen verwendet werden kann und die rechte Maustaste durch einen Doppeltipp auf den Bildschirm ausgelöst wird (dieses Verhalten kann per Code geändert werden).

Gesten

Dieses Ereignis ist dasjenige, das ausgelöst wird, wenn der Benutzer den Bildschirm berührt (auf mobilen Geräten) oder klickt und die Maus bewegt (auf allen anderen Plattformen). Diese Ereignisse sind den Mausereignissen insofern ähnlich, als dass es reguläre Versionen und globale Versionen gibt. Die regulären Versionen dieser Ereignisse werden nur ausgelöst, wenn die Berührungen auf einer Instanz stattfinden, die ein Sprite (oder eine Maske) hat und die Berührungen innerhalb ihrer Bounding Box stattfinden. Die globalen Versionen dieser Ereignisse werden jedoch ausgelöst, wenn der Benutzer eine beliebige Stelle auf dem Bildschirm berührt.

Die Gestenereignisse erkennen Folgendes:

  • taps - wenn ein Benutzer schnell klickt/berührt und loslässt
  • zieht - wenn ein Benutzer berührt/klickt und es beibehält und dann seinen Finger/Cursor bewegt
  • flicks - wenn der Benutzer einen Touch/Klick in einer Bewegung bewegt und loslässt
  • pinches - wenn der Benutzer zwei Finger auf dem Bildschirm hat und sie dann zusammen/ auseinander bewegt
  • rotiert - wenn der Benutzer zwei Finger auf dem Bildschirm hat und sie dann um einen Punkt dreht

Die verschiedenen Ereignisse enthalten immer eine DS-Map mit der Bezeichnung "event_data", die eine Reihe von Schlüssel/Wert-Paaren mit Daten über die Berührungs-/Klickposition und Bewegung enthält. Zeichnen

Diese Ereigniskategorie ist diejenige, die bestimmt, was Sie auf dem Bildschirm sehen, wenn Sie Ihr Spiel ausführen, und ist in verschiedene diskrete Ereignisse unterteilt:

Wie Sie sehen können, hat die Ereignis-Kategorie "Zeichnen" mehrere verschiedene Ereignistypen. Draw Begin, Draw und Draw End sind die "Standard"-Zeichenereignisse, die Sie wahrscheinlich am häufigsten verwenden werden. Standardmäßig wird das Hauptereignis "Draw" immer für jede Instanz aufgerufen, unabhängig davon, ob sie ein Sprite hat oder nicht, obwohl, wenn Sie die Instanz als unsichtbar kennzeichnen, das Ereignis nicht ausgelöst wird (denken Sie also daran, wenn Sie Spiellogik im Draw-Ereignis eines unsichtbaren Objekts haben, da diese nicht ausgeführt wird). Das Haupt-Zeichenereignis ist auch der Ort, an dem GameMaker Studio 2 das Instanz-Sprite standardmäßig zeichnet, wenn weder Code noch Aktionen im Ereignis gibt (d. h.: Sie haben es nicht in die Ereignisliste für das Objekt aufgenommen). Das Standard-Zeichnen verwendet das Sprite, das mit der Instanz verknüpft ist, und zeichnet dieses mit allen Transformationen, die im Code gesetzt sind, oder mit angewendeten Aktionen.

Die normalen Draw-Ereignisse zeichnen vor den Draw-GUI-Ereignissen und zwischen den Pre-Draw- und Post-Draw-Ereignissen , was bedeutet, dass alles, was in diesem Ereignis gezeichnet wird, unter dem des Draw-GUI-Ereignisses gezeichnet wird, unabhängig von der Ebene (d. h.: alles, was im Draw-GUI-Ereignis gezeichnet wird, wird immer über allem gezeichnet, was im normalen Draw-Ereignis gezeichnet wird, unabhängig von der Ebenenreihenfolge).
