Code kommentieren

Bei der Zusammenarbeit mit anderen Personen, bei der Bearbeitung eines großen Projekts oder auch nur für Ihre eigene Referenz und Fehlersuche ist das Hinterlassen von Kommentaren in Ihrem Event-Code und Ihren Scripts sehr wichtig. In GameMaker Studio 2 haben Sie verschiedene Mechanismen die Ihnen helfen, Notizen und Kommentare zu Codeabschnitten und sogar ganzen Codeblöcken zu hinterlassen, die in der Ereignisliste des Objekts zu sehen sind, das Sie gerade bearbeiten. Auf diese Weise können Sie Notizen für sich selbst und für Ihre Kollegen hinterlassen, oder um ein besonders kniffliges Stück Code zu erklären, oder auch nur eine Erinnerung zu hinterlassen, was etwas tut oder was Sie in Zukunft tun wollen.

Um einen einfachen einzeiligen Kommentar zu hinterlassen, können Sie als erstes // vor dem Text verwenden. Zum Beispiel:

//Initialisieren Sie Variablen
sr = 10;
hp = 100;

Sie können auch mehrzeilige Kommentare hinterlassen, um Lob auszusprechen, einen kompletten Codeabschnitt zum Debuggen auszulassen oder auch um Hinweise zu hinterlassen, wie Sie den Code in Zukunft ändern wollen. Dafür können Sie /* ... */ wie dieses:

/*
GIF-Variablen auf Anfangswerte setzen (diese können bearbeitet werden, so dass Sie die Setter-Funktionen nicht ständig verwenden müssen, wenn Sie immer die gleichen Werte verwenden)

WICHTIG! Bitte sehen Sie sich die einzelnen "set"-Skripte an, um ausführlichere Informationen über die verschiedenen Variablen zu erhalten.
*/

frame_num = 30;
frame_steps = 5;
frame_width = 256;
frame_height = 256;
frame_quality = gifcam_quality_high;

Abgesehen von Kommentaren können Sie auch spezielle #region-Tags verwenden, um einen Codeabschnitt für Code Folding zu markieren, und jede Region kann einen Kommentar enthalten. Das funktioniert so, dass Sie ein #region-Tag - zusammen mit einem beliebigen Kommentartext - an den Ein #endregion-Tag wird - zusammen mit einem beliebigen Kommentartext - am Anfang eines Codeabschnitts eingefügt und ein entsprechendes #endregion-Tag am Ende des Abschnitts, so dass der Code "gefaltet" werden kann, d. h. er wird versteckt, um die Anzahl der im Editor sichtbaren Codezeilen zu reduzieren einer Zeit:

Der aktuelle Code sieht in etwa so aus:

#region Dies ist eine Beispielregion

width = sprite_get_width(sprite);
height = sprite_get_height(sprite);
xoff = sprite_get_xoffset(sprite);
yoff = sprite_get_yoffset(sprite);

#endregion Hier ggf. weiterer Kommentar

Abschließend sei noch erwähnt, dass Sie bei der Arbeit mit Ereignissen im Objekteditor einen speziellen "description" Kommentar am Anfang des Ereignisses verwenden können, um zu sagen was der allgemeine Inhalt des Ereignisses ist, und dies wird im Ereignisfenster des Objekteditors angezeigt. Dieser sollte in der allerersten Zeile des Code-Editors stehen und ist wie folgt formatiert:

/// @description <Dies ist eine Ereignisbeschreibung>

Dies wird nun im Ereignisfenster angezeigt, z. B:

Beachten Sie, dass Sie in den Voreinstellungen des Objekteditors einen Standardkommentar festlegen können, der für alle Ereignisse erstellt wird.

Wenn Sie Skriptfunktionen schreiben und steuern möchten, wie diese von der IDE behandelt werden (z. B. automatische Vervollständigung oder Anzeige der Skriptparameter), sollten Sie Ihre Kommentare in der JSDoc-Notation schreiben, wie hier erklärt.
